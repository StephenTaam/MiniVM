STT Metadata VM 从零使用说明

这个页面把项目的使用方式讲成一条完整路径：先编译运行，再理解 CodeBlock JSON 数据结构，最后按指令集写 metadata 程序并调试。

USAGE.md LANGUAGE_VM_GUIDE.md METADATA_JSON.md INSTRUCTION_SET.md SOURCE_COMPAT.md 当前输入：静态 JSON，不是 .ose 源码

1. 这是什么

STT Metadata VM 是一个训练版 bytecode 虚拟机。它不直接解析 `.ose` 源码，也不读取二进制 RSE 文件；它读取一份静态 CodeBlock metadata JSON，然后执行里面的栈式指令。

伪代码或外部编译器
        ↓
CodeBlock metadata JSON
        ↓
STT Metadata VM 执行

原始资料里最重要的约束是 CodeBlock 数据结构和指令名/opcode。本项目按原始资料保留 `type`、`cbname`、`constants`、`names`、`blocks`、`instructions`、`exceptiontable` 等字段；指令也按原始资料尽量对齐，宿主相关行为用训练版语义模拟。

2. 快速运行

macOS / Linux / WSL

make
build/stt_vm run examples/minimal.json

预期输出会看到 `6`，对应示例里的 `3 * 2`。

Windows / 通用 CMake

cmake -S . -B build-cmake
cmake --build build-cmake
build-cmake\Debug\stt_vm.exe run examples\minimal.json

Visual Studio 多配置构建下，Release 可执行文件通常在 `build-cmake\Release\stt_vm.exe`。

命令格式

stt_vm run <metadata.json> [--trace] [--dump-tables] [--entry name]
stt_vm dump <metadata.json>
stt_vm disasm <metadata.json>
命令 用途 常用场景
run 加载 JSON 并执行 VM。 确认程序能跑、观察输出。
dump 打印 Program、CodeBlock、常量表、名字表、子块、指令和扩展字段。 检查 JSON 静态结构有没有写对。
disasm 当前和 dump 输出接近，重点看每条指令的 oparg 解释。 确认 `arg` 查的是常量表、名字表还是子块表。
--trace 逐条指令打印栈、locals、globals 和查表结果。 调试栈式程序。

3. 执行模型

一个 JSON 程序由 `Program` 外壳和 `root` CodeBlock 组成。CodeBlock 里最重要的是四张表：常量表、名字表、子块表和指令表。

Program
  version
  entry
  root: CodeBlock

CodeBlock
  constants       字面量和静态值
  names           变量名、函数名、属性名
  blocks          函数、类、Neta 等子 CodeBlock
  instructions    栈式指令序列

VM 是栈式执行。大多数指令从栈顶取值，再把结果压回栈。文档里用右侧表示栈顶：

LOAD_CONST 0
before: [...]
after:  [..., constants[0]]

BINARY_OP 1
before: [..., left, right]
after:  [..., left + right]

4. JSON 字段

Program 外壳

字段 类型 说明
version number metadata 版本号，当前示例使用 `1`。
entry string 默认入口名。`run --entry name` 可以覆盖。
root object 顶层 CodeBlock。没有源码 parser 时，这是 VM 的主要输入。

CodeBlock 类型

type 资料类名 用途
0 CodeBlock / Script 顶层脚本块。
1 FuncCodeBlock 函数块，使用 `argnames`、`defaults`、`flags` 等函数字段。
2 ClassCodeBlock 类块，使用 `inherits`、`syntactic`、`generics` 等类字段。
4 NetaCodeBlock Neta 块，当前用训练版执行语义。

CodeBlock 字段总表

字段 类型 怎么用
type number 块类型，规范写法是 `0/1/2/4`。字符串 type 只作为兼容写法。
cbname string 块名称，用于 trace frame 名、函数名、类名、模块名等展示。
constants array 常量池，`LOAD_CONST arg` 读取 `constants[arg]`。
names / name array<string> 名字表，变量名、函数名、属性名和作用域名字都放这里。`name` 是兼容别名。
blocks array<CodeBlock> 嵌套子块。`DEFINE_FUN 0`、`DEFINE_CLASS 0` 这类指令会读取 `blocks[arg]`。
instructions array 指令序列。推荐写 object 数组，例如 `{ "op": "LOAD_CONST", "arg": 0 }`。
exceptiontable array 异常处理表。`THROW` 会按 `startIp/endIp/handlerIp` 查 handler，`CATCH` 取 pending error。
instlnums array<number> 每条指令对应源码行号，可为空。用于 dump/trace 辅助定位。
baseline number 起始行号偏移，可为空或 `0`。
currline number 训练版保留字段，资料核心表没有；当前保存并 dump。
argnames array<string> 函数参数名，只对 FuncCodeBlock 有意义。
defaults array<Value> 函数参数默认值。
flags array<string> 函数类型标记，例如资料里出现的 `dict`。
inherits array<string> 类继承列表，只对 ClassCodeBlock 有意义。
annotations array 注解列表。当前保留并可通过注解相关指令构造 metadata。
syntactic object 或空数组 类扩展特性列表。资料表写 Dict；loader 兼容 `{}` 和 `[]`。
generics array<string> 泛型参数名或泛型列表。

5. 指令格式

推荐使用字符串 `op`，因为调试时可读性最好：

{ "op": "LOAD_CONST", "arg": 0 }

也支持资料里的数字 opcode 和 `oparg` 字段：

{ "opcode": 25, "oparg": 0 }

`25` 是十进制，对应资料里的 `0x19 LOAD_CONST`。JSON 标准不支持 `0x19` 这种字面量，所以文件里写十进制。

oparg 怎么解释

类型 含义 例子
ConstIndex 查 `constants[arg]`。 LOAD_CONST 0
NameIndex 查 `names[arg]`。 LOAD_NAME 2
BlockIndex 查 `blocks[arg]`。 DEFINE_FUN 0
ArgCount 调用参数数量。 CALL 2
BinaryOp / UnaryOp / CompareOp 操作符枚举。 BINARY_OP 1 表示加法。
JumpTarget / JumpOffset 跳转目标或偏移。也可以额外写 `target` 字段。 { "op": "POP_JUMP_IF_FALSE", "target": 10 }
ImmInt 立即数整数。 LOAD_IMM8 7

调试时加 `--trace`，VM 会打印 `oparg kind`、查表位置、查表结果、栈变化和 locals/globals。看懂 trace，基本就能自己排查大多数 JSON 指令错误。

6. 怎么写程序

先写静态表。

字面量放 `constants`，变量名/函数名/属性名放 `names`，函数或类子块放 `blocks`。

再写栈式指令。

想赋值就先把值压栈再 `STORE_NAME`；想计算就先压两个值再 `BINARY_OP`；想调用函数就先压函数、再压参数、最后 `CALL 参数个数`。

最后用 dump 和 trace 验证。

先 `dump` 看 `arg` 有没有查对表，再 `run --trace` 看每一步的栈是否符合预期。

最小例子：a = 3; b = 2; echo a * b

{
  "version": 1,
  "entry": "main",
  "root": {
    "type": 1,
    "cbname": "main",
    "constants": [
      { "type": "int", "value": 3 },
      { "type": "int", "value": 2 }
    ],
    "names": ["a", "b", "c"],
    "blocks": [],
    "instructions": [
      { "op": "LOAD_CONST", "arg": 0 },
      { "op": "STORE_NAME", "arg": 0 },
      { "op": "LOAD_CONST", "arg": 1 },
      { "op": "STORE_NAME", "arg": 1 },
      { "op": "LOAD_NAME", "arg": 0 },
      { "op": "LOAD_NAME", "arg": 1 },
      { "op": "BINARY_OP", "arg": 3 },
      { "op": "STORE_NAME", "arg": 2 },
      { "op": "LOAD_NAME", "arg": 2 },
      { "op": "ECHO", "arg": 0 },
      { "op": "RETURN_VALUE", "arg": 0 }
    ],
    "exceptiontable": [],
    "instlnums": [],
    "baseline": 0
  }
}
build/stt_vm run examples/minimal.json --dump-tables --trace

7. 指令分类

当前 registry 中的指令都已经接入执行分支。下面按用途分组，方便快速定位；完整逐条解释看 INSTRUCTION_SET.md，资料 opcode 数值看 SOURCE_COMPAT.md。

常量和值 LOAD_CONSTLOAD_IMM8LOAD_NONEOPCODE_NUM 压入常量、立即数、空值或 opcode 数量。
变量和作用域 LOAD_NAMESTORE_NAMELOAD_GLOBALSTORE_VARIABLEUNLET_VARIABLE 维护 locals、globals、const/env/type 变量和带作用域名字。
函数和调用 DEFINE_FUNLOAD_FUNCTIONCALLEXTENDED_ARGRETURN_VALUE 从 `blocks` 创建函数，支持普通参数、命名参数、展开参数、defer 和 tennis call。
算术和比较 BINARY_OPUNARY_OPCOMPARE_OPTO_BOOLTYPE_CAST 执行数值、字符串、bool 和比较操作。未知枚举会报清晰错误。
容器 BUILD_LISTBUILD_DICTLOAD_SUBSCRSTORE_SUBSCRSLICE 构造 list/dict，支持索引、负索引、切片、解包和删除下标。
跳转和循环 JUMP_FORWARDPOP_JUMP_IF_FALSEFOR_ITERBREAKCONTINUE_LOOP 支持目标 ip、相对偏移和 iterator 循环。
类和对象 DEFINE_CLASSCREATE_INSTANCELOAD_ATTRSTORE_ATTRINSTANCE_OF 创建训练版 class/instance，读写属性，模拟继承和实例判断。
模块、脚本和 FQN DECLARE_PACKAGELOAD_MODULESOURCESET_FQNGET_PATH_BY_FQN 维护本地模块表、脚本表、包名、路径映射。不会执行真实系统 shell。

编译器书签 ENTER_LOOPLEAVE_LOOPEXIT_SCOPEGARBAGE_COLLECT 作为 no-op/bookkeeping 处理，用来保留资料里的编译器结构信息。

9. 调试方法

推荐顺序是：先静态检查，再跑 trace，再根据错误上下文修 JSON。

先看结构。 
build/stt_vm dump examples/your_file.json
再看执行。 
build/stt_vm run examples/your_file.json --dump-tables --trace
最后看错误上下文。

RuntimeError 会带 frame、ip、instruction、oparg kind、lookup、stack before/current、locals 和 globals。大多数错误都能从这里定位到是哪条指令的 `arg` 查错表、栈上少值或变量名不存在。

如果 `LOAD_CONST 0` 报错，先看 `constants` 是否为空；如果 `STORE_NAME 2` 报错，先看 `names[2]` 是否存在；如果 `CALL 2` 报错，先看栈上是不是按“函数、参数1、参数2”的顺序准备好了。

10. 边界和兼容

能力 状态 说明
CodeBlock 数据结构 已对齐 使用资料里的 `0/1/2/4` 类型和静态字段。字符串 type 只是兼容。
资料 opcode 已注册已执行 原始资料中可识别 opcode 已登记数值；registry 内指令都有 VM 分支。
宿主相关指令 训练语义 `SOURCE`、`EXECUTE`、模块、FQN、注解、泛型等使用本地 metadata 语义，不执行真实 Harmony/Neta 宿主副作用。
.ose 源码 parser 未包含 当前要手写 JSON，或者未来由外部编译器生成同样的 JSON。
二进制 RSE 未包含 当前不读写 RSE，只读 JSON。
未知 opcode 执行时报错 dump/disasm 可以看到结构；真正执行未知或 `OPCODE_ILL` 会报 RuntimeError。

最后更新目标：让一个没有看过源码的人，也能按这个页面完成构建、运行、改 JSON、查字段、查指令和调试。
